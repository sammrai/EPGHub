# Match-report queue processor

Drains the open `fix-match` issue queue created by the in-app
"マッチを報告" button. Each issue carries a `programs.id` + raw EPG
title + the user's TVDB-mapping concern. This skill is the auto-pilot:
take one issue, run the `fix-episode-match` workflow inside an
isolated sub-agent, and resolve the issue (close or re-label).

The user explicitly authorised commit + close actions inside this
skill — that is the *only* place the standing "勝手にコミットしない"
rule is suspended. Outside this skill, do not commit.

## Key invariants

1. **One issue per pass.** The processor is sequential; concurrent
   sub-agents would race on `matchService.ts` edits. `/loop` reruns
   the skill, the skill picks one issue, finishes it, returns.
2. **Sub-agent isolation.** The fix-episode-match diagnosis runs in a
   clean-context sub-agent so the orchestrator's history doesn't bias
   the verdict. Pass the issue body verbatim plus the inference
   rubric below.
3. **Default branch commit closes the issue** via the keyword
   `Fixes #N` in the commit message. PR is not required for solo
   maintainer flow. (`close[s|d]` / `fix[es|ed]` / `resolve[s|d]`
   all work.)
4. **`needs-review` is the safety valve.** When the sub-agent can't
   confidently diagnose or its fix fails the test suite, the issue
   gets labelled `needs-review` and stays open. Never close on
   uncertainty — the user's standing rule is honest confidence.

## Issue lifecycle

```
[open + fix-match]
    │
    ├─ auto-pilot diagnoses ─→ [closed: completed] (commit Fixes #N)
    │
    └─ inference fails ─→ [open + fix-match + needs-review]
                              │
                              │ ← human/user adds comment with hint
                              │ ← human removes `needs-review` label
                              ▼
                          [open + fix-match]   ← back in queue
                              (auto-pilot retries, this time
                               reading the new comments for context)
```

The auto-pilot does **not** remove `needs-review` itself. The human
removes the label when they've left a clarifying comment, and that
re-enters the issue into the queue. This keeps the human as the
final arbiter — auto-pilot never silently retries a deferred case.

**How a human un-defers an issue**:

```sh
# 1. Add an explanatory comment (expected S/E, broadcaster hint, etc.)
gh issue comment <N> --body "期待値: S2E5 / 理由: …"

# 2. Remove the label so the auto-pilot's filter picks it up again
gh issue edit <N> --remove-label needs-review
```

Or via the GitHub web UI: open the issue, click the `needs-review`
chip in the Labels sidebar to remove it, post a comment from the
text box.

Removing the label without leaving a comment is allowed but rarely
useful — the auto-pilot's inference rubric will produce the same
result as last time and likely re-defer. The comment is what
changes the outcome.

## Step 1: pick the next issue

```sh
gh issue list \
  --label fix-match \
  --state open \
  --json number,title,body,comments,labels \
  --limit 50
```

Filter client-side: drop any issue whose `labels[].name` includes
`needs-review`. Pick the **oldest remaining** issue (FIFO) — that's
the one a human most likely just unblocked. If nothing remains, the
queue is drained — report "queue drained, N issues processed this
run" and exit. Under `/loop` the harness schedules the next wakeup.

## Step 2: parse program context from the issue body + comments

**Issue body** (the original report — Markdown form generated by
`Modal.tsx`). Pull:

- `programs.id` → from `- programs.id: \`<id>\``
- `title` → from `- title: \`<raw>\``
- `ch`, `startAt` → same shape
- Current `tvdb_id` / `season` / `episode` → from `**現在のマッチ**`
  block. `null`/`—` means unmatched.
- Expected outcome → from `**期待される結果**` block. Often empty —
  see Step 3 inference rules.

**Comments** (added when the issue was previously deferred to
`needs-review`). Pull `comments[].body`. The human comment will
typically contain:

- Explicit expected outcome (`tvdb_id=425870, S2E5` / 「マッチしない
  のが正しい」 / 「これは○○の特殊放送なので…」).
- A correction to the auto-pilot's previous misdiagnosis ("前回の
  推測は間違い、実際は X が正解").
- A hint about the broadcaster pattern that was missed.

Treat the **most recent human comment** as authoritative. It
overrides Step 3 inference. If multiple comments exist, the latest
one wins — earlier comments may have been from the auto-pilot's own
previous deferral message.

The auto-pilot's own previous comments (recognisable by the prefix
`Auto-pilot deferred:` or by the GitHub username matching the
runtime user) should be skipped when looking for human input.

## Step 3: infer the expected outcome (best-effort, never bail)

The orchestrator does **best-effort inference** to enrich the
sub-agent's prompt — it does **not** decide whether the issue is
solvable. That call belongs to the sub-agent, which has clean
context and runs the full diagnosis. **needs-review is a last
resort, not a routing default.** Always spawn the sub-agent.

**If the latest human comment from Step 2 explicitly states an
expected outcome, skip the rubric entirely** — pass the comment as
the authoritative expected outcome.

Otherwise apply this rubric in order, picking the first that fits:

1. **Title cleanly matches a real anime/drama on TVDB?** Run
   `node --import tsx scripts/verify-match.ts '<programs.id>'` to see
   what the current code resolves to. If `findEpisodeForProgram`
   returns a coherent `{s,e,name}` whose `name` matches a subtitle
   inside the EPG title (or whose `e` matches a `#N`/`第N話`/`（NN）`
   marker in the title), **the user's likely concern is "this should
   match a real TVDB episode"** — proceed with that as the expected
   outcome.
2. **`tvdb_id = null` and the title is clearly a real series**
   (anime/drama/movie name + episode marker), **expected =
   "matched to a TVDB entry"**. Don't pre-decide which entry; let
   `enrichUnmatched` resolve via the matcher's normal path.
3. **Title is a digest / 総集編 / 特別編 / ダイジェスト / 傑作選 /
   オープニング / 番宣 / SP の単発**, **expected = "should not
   match a numbered episode"** (S/E null is correct).
4. **Title carries `（NN）` zenkaku parens or `第N話` and the cached
   episode list spans seasons** → expected is the episode at that
   index per the cumulative-N walk. Verify against the cache.
5. **Otherwise → no inference reached.** Pass that fact to the
   sub-agent verbatim (`"no inference reached — diagnose what would
   be a coherent expected result"`). The sub-agent decides whether
   to proceed with its own diagnosis or to apply `needs-review`. The
   orchestrator does not pre-bail.

## Step 4: hand off to the fix-episode-match sub-agent

Spawn an Agent (general-purpose, clean context) with a self-contained
prompt that includes:

- The issue number `#N` (so the sub-agent embeds it in commit
  messages).
- The parsed program info (id, title, ch, startAt, current TVDB).
- The inferred expected outcome from Step 3 (or "no inference
  reached — diagnose what would be a coherent expected result").
- Explicit instruction: **run the `fix-episode-match` skill end to
  end**, including verify-match.ts.
- Authorisation: **may commit code + golden fixture changes**, but
  must include `Fixes #N` in the commit subject so the issue auto-
  closes on push. **Do NOT push** — leave the commit local.
- Authorisation: **may run `gh issue close N` / `gh issue
  edit N --add-label needs-review`** as appropriate.

The sub-agent reports back one of three outcomes:

- **Fixed**: code change committed, `verify-match` passes, all skill
  tests green, issue gets auto-closed by the commit (when pushed) or
  the sub-agent closes it explicitly with a comment referencing the
  commit SHA.
- **Already correct (stale data only)**: no code change, sub-agent
  closes the issue with a comment explaining "current code resolves
  correctly; the DB had stale S/E from a removed fallback".
- **Needs review**: diagnosis ambiguous, expected outcome not
  inferable, or test suite fails after attempted fix. Sub-agent
  applies `needs-review` label and leaves issue open.

## Step 5: post-handoff verification (orchestrator side)

Don't trust the sub-agent's self-report blindly. After it returns:

1. Check `gh issue view N --json state,labels` — confirm the issue
   is in the state the sub-agent claimed.
2. If sub-agent reported "Fixed", verify with `git log -1
   --format='%s%n%b'` that the latest commit's message contains
   `Fixes #N` (or equivalent keyword). If missing, fix the commit
   message (rebase or amend with the keyword) so the auto-close
   actually fires when pushed.
3. Run the four-command Step 5 suite from `fix-episode-match`
   (`findEpisodeForProgram.test.ts`, `matchService.test.ts`,
   `typecheck`, `test:golden`) one more time as a final guard —
   sub-agent runs are sometimes incomplete.

If any verification fails, **revert the sub-agent's commit**
(`git reset --hard HEAD~1` — only if the commit is local, never if
pushed) and apply `needs-review` to the issue with a comment quoting
the failure. Don't ship a half-fix.

## Step 6: needs-review fallback (the universal safety valve)

The `needs-review` label is the catch-all for:

- Sub-agent declined to infer the expected outcome.
- Verify-match.ts returned a result that doesn't match any plausible
  inference.
- Code change was made but tests failed.
- Orchestrator's post-handoff verification (Step 5) caught a
  discrepancy.
- This is the **N-th** time the same issue cycled through (auto-pilot
  retried after a human comment but still couldn't resolve). After
  ~2 cycles of round-tripping, escalate explicitly in the comment so
  the human knows the auto-pilot has given up rather than being
  about to retry yet again.

To apply (idempotent — `gh` no-ops if already labelled):

```sh
gh issue edit N --add-label needs-review
gh issue comment N --body "Auto-pilot deferred: <one-line reason>.
期待値を本 issue にコメント → \`needs-review\` ラベルを外すと再処理
されます。"
```

The comment template tells the human exactly how to re-enter the
issue into the queue. Copy that wording — it's the user-facing
contract for the lifecycle described above.

The label has to exist on the repo first. Create it once:

```sh
gh label create needs-review \
  --color D93F0B \
  --description "auto-pilot couldn't resolve; human review required" \
  || true
```

## Step 7: report and loop

End-of-pass output (one short paragraph):

- Issue number processed.
- Outcome (Fixed / Already-correct / Needs-review).
- One-line reason or commit SHA.

If invoked under `/loop`, return so the harness schedules the next
wakeup. Otherwise just exit; the user can re-run.

## Things to avoid

- **Pushing commits**. Leave them local; the user pushes when they
  trust the batch. (`git push` is outside this skill's scope.)
- **Closing issues without a clear resolution comment.** Every close
  needs a one-line "why" — even "stale data, current code returns
  null/correct value".
- **Touching issues that aren't in the `fix-match` label.** This
  skill's scope is the match-report queue only.
- **Concurrent sub-agents.** Always sequential — one issue, one
  commit, then the next. Repo state must be clean before the next
  pass.
- **Editing the issue body.** It's the user's report; preserve it
  verbatim. Add comments instead.
- **Closing with `not planned` reason.** Use the default `completed`
  reason for fixes; leave `not planned` for the human.
