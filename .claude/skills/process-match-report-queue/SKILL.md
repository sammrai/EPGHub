# Match-report queue processor

Drains the `fix-match` issue queue created by the in-app "マッチを報告"
button. Each issue carries `programs.id`, raw EPG title, and the user's
TVDB-mapping concern. This skill is the auto-pilot: take one issue, run
`fix-episode-match` in a clean-context sub-agent, and resolve the issue
(close or re-label).

This is the only context where the standing "勝手にコミットしない" rule
is suspended — the user explicitly authorised commit + close inside this
skill. Outside, do not commit.

## Key invariants

1. **One issue per pass, sequential.** Concurrent sub-agents would race
   on `matchService.ts`. Process one issue end-to-end, post-verify, then
   loop to the next.
2. **Sub-agent isolation, foreground.** Spawn the diagnosis sub-agent
   in clean context (no orchestrator history) and **synchronously** —
   never `run_in_background`. Step 5 verification must execute in the
   same turn so a bad commit can be reverted before continuing.
3. **Explicit close, not auto-close.** Sub-agents close resolved issues
   directly with `gh issue close N`. The `Fixes #N` keyword in the commit
   body preserves the cross-reference at push time but is **not**
   responsible for state transition; relying on it would leave issues
   lingering in the FIFO queue until push.
4. **`needs-review` is the safety valve.** When diagnosis is ambiguous
   or a fix fails the test suite, label `needs-review` and leave the
   issue open. Never close on uncertainty.

## Issue lifecycle

```
[open + fix-match]
    │
    ├─ auto-pilot resolves ─→ [closed]   (commit Fixes #N + explicit close)
    │
    └─ inference fails ─→ [open + needs-review]
                              │ ← human comments hint + removes label
                              ▼
                          [open + fix-match]   (re-enters queue)
```

The auto-pilot never removes `needs-review` itself — the human is the
final arbiter. To un-defer:

```sh
gh issue comment <N> --body "期待値: …"
gh issue edit <N> --remove-label needs-review
```

## Step 1: pick the next issue

```sh
gh issue list --label fix-match --state open \
  --json number,title,body,comments,labels --limit 50
```

Filter client-side: drop `needs-review`-labelled issues. Pick the
**oldest remaining** (FIFO). Also skip any issue whose number already
appears as `Fixes #N` in `git log origin/main..HEAD --grep "Fixes #<N>"`
(would produce a duplicate commit).

If nothing remains → queue drained. Report and exit (no continuation).

## Step 2: parse program context

**Issue body** (Markdown form generated by `Modal.tsx`):
- `programs.id`, `title`, `ch`, `startAt` from the labelled fields.
- Current `tvdb_id` / `season` / `episode` from `**現在のマッチ**`
  (`null`/`—` = unmatched).
- Expected outcome from `**期待される結果**` (often blank).

**Comments**: when the issue was previously deferred, the human leaves
a clarifying comment (expected S/E, broadcaster hint, correction). The
**most recent human comment overrides Step 3 inference**. Skip comments
authored by the auto-pilot (prefix `Auto-pilot deferred:` / matching
runtime user).

## Step 3: infer the expected outcome (best-effort, never bail)

Best-effort enrichment of the sub-agent's prompt — the orchestrator
does **not** decide solvability. **needs-review is a last resort, not a
routing default — always spawn the sub-agent.**

If a human comment from Step 2 states an expected outcome, pass it
verbatim. Otherwise apply this rubric and pick the first that fits:

1. **Title looks like a real anime/drama on TVDB?** Run
   `node --import tsx scripts/verify-match.ts '<programs.id>'` to see
   what current code resolves to. If the result has a coherent
   `{s,e,name}` matching a marker in the title, the user's likely
   concern is "this should match a real TVDB episode."
2. **`tvdb_id = null` and the title is clearly a real series** →
   expected = "matched to a TVDB entry" (let the matcher pick which).
3. **Title is a digest / 総集編 / 特別編 / ダイジェスト / 傑作選 /
   オープニング / 番宣 / SP の単発** → expected = "should not match a
   numbered episode" (S/E null is correct).
4. **Title has `（NN）` zenkaku parens or `第N話` and the cache spans
   seasons** → expected = the cumulative-N walk index. Verify against
   the cache.
5. **Otherwise** → pass `"no inference reached — diagnose what would
   be a coherent expected result"`. The sub-agent decides whether to
   proceed or apply `needs-review`.

## Step 4: hand off to the fix-episode-match sub-agent

Spawn a general-purpose Agent (clean context, **foreground**) with a
self-contained prompt: issue number, parsed program info, inferred
expected outcome, instruction to run `fix-episode-match` end-to-end
(including verify-match.ts), and authorisation to commit (with
`Fixes #N` in the subject), close the issue, or apply `needs-review`.
Do NOT push.

**Mandatory clause in the sub-agent prompt**: commit only the lines
you intentionally edited. Stage by explicit path (no `-A` / `.` /
`-a`); when a file already has user changes, hunk-stage with
`git add -p`.

The sub-agent reports back **one of four outcomes**:

- **Fixed**: code change committed, all tests + verify-match pass.
  Close with `gh issue close N --comment "Fixed in commit <sha>:
  <one-liner>"`.
- **Already correct (stale data only)**: `verify-match.ts` returns the
  right answer; DB has stale S/E. **No code change.** Close with a
  comment explaining the stale-data theory.
- **Fixed by side-effect of another commit** *(during the same drain
  run)*: an earlier commit already covers this issue (e.g. a
  `BLOCK_PREFIXES` regex absorbing a class). **No new commit.** Close
  with `gh issue close N --comment "Already covered by commit <sha>
  (issue #M): <dependency explanation>"`. The cross-reference is what
  distinguishes this from "stale data".
- **Needs review**: ambiguous diagnosis or test failure. Apply
  `needs-review` and leave the issue **open**.

## Step 5: post-handoff verification

After the sub-agent returns:

1. `gh issue view N --json state,labels` — outcome states should be
   `CLOSED` (Fixed / Already-correct / Side-effect) or `OPEN` with
   `needs-review` (Needs review).
2. `git show HEAD --stat` — commit should only contain files the
   sub-agent intentionally edited. If unrelated paths appear (user's
   in-flight `app/` work, etc.), the sub-agent bundled them in →
   rollback required.
3. Re-run the `fix-episode-match` four-command suite.

**On failure**: `git reset --soft HEAD~1`, drop unintended files with
`git restore --staged`, recommit. Apply `needs-review` (reopen the
issue first if it was closed).

## Step 6: needs-review fallback

Triggered when: sub-agent declines, verify-match disagrees with any
plausible inference, tests fail, or Step 5 catches a discrepancy.

```sh
gh issue edit N --add-label needs-review
gh issue comment N --body "Auto-pilot deferred: <one-line reason>.
期待値を本 issue にコメント → \`needs-review\` ラベルを外すと再処理
されます。"
```

Copy the comment wording verbatim — it's the user-facing contract for
how to un-defer (Issue lifecycle diagram).

The label must exist on the repo. Create once if missing:

```sh
gh label create needs-review --color D93F0B \
  --description "auto-pilot couldn't resolve; human review required" || true
```

## Step 7: report and continue inline

End-of-pass output (one short paragraph): issue number, outcome,
one-line reason or commit SHA.

**Default: continue inline** — when issues remain, loop back to Step 1
in the same conversation turn. Sleeping between passes is pure overhead
while the user is actively driving the session.

**Terminate (no continuation) when**:
- Step 1 returned no remaining issues (queue drained).
- The user explicitly told you to stop this turn.

If you genuinely need to step out of the conversation (rare for a
foreground queue-drain), use `ScheduleWakeup` with
`prompt: "/loop /process-match-report-queue"` — the `/loop` prefix is
mandatory or the bare slash-command lands as `Unknown command`.

## Things to avoid

- **Pushing commits.** Leave local; user pushes when they trust the batch.
- **Closing without a resolution comment.** Every close needs a one-line
  "why". Side-effect closes **must** reference the source commit SHA +
  issue number.
- **Touching issues outside the `fix-match` label.** Scope discipline.
- **Background sub-agents** (`run_in_background`). Foreground only —
  Step 5 must run in the same turn.
- **Concurrent sub-agents.** Always sequential. Stage only `server/`
  paths in your commit (the user's in-flight `app/` work stays out).
- **`git add -A` / `git add .` / `git commit -a`.** Forbidden — sweeps
  up user in-flight work. Stage by explicit path; `git add -p` for
  files with mixed edits.
- **Editing the issue body.** Preserve verbatim; use comments instead.
- **Closing with `not planned` reason.** Use the default `completed`
  for fixes; leave `not planned` for the human.
